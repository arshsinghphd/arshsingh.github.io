                  00_Introduction to Flow Optimization

                  In this set of wikis, I describe an approach to the classical problem of optimizing the flow from a source to a sink through a Flow Network. I have written the solutions in Python. I will describe and explain parts of the code and show how to put them together.

                  Introduction

                  Let's start by clearing some terms: a definition of a flow network follows soon, but for an intuition, visualize a network of an assortment of pipes connected via multi-sized couplings. There is only one inlet, say a pump, and one outlet, maybe a large container, a sump. We are given the capacity of each pipe (based on its diameter, thickness of the walls, material of construction etc.). The task is to set the pump such that we flow the maximum amount of liquid through the system, no more no less. If we try to flow more than max. possible, we will disrupt the system: causing leakages at joints, burst pipes etc.

                  It may seem like a simple problem at the first glance, but is not. Later in this wiki, playing around with a toy example, I will show how, if we use brute force this seemingly simple problem has exponential time-complexity of at least O[2m], where m is the number of pipes/edges in the network. Even then, we may not converge to the correct solution.

                  If you are not familiar with time complexity, for now you can understand it as follows: as the number of edges increase from 2 to 8 (4 times), the number of operations needed to solve the problem go from 4 to 256 (64 times). If the edges increase to 32, the brute force solution will not be able to finish this search on 512 MB RAM, running into memory issues.

                  This problem has variations for many seemingly unrelated real world problems. One of these is circulation in a network, or the minimum flow problem which is used in assembling genome from small fragments of DNAs.

                  While there are many more elegant solutions to the Maximum flow problem, closer to how a human would visually look for a path, it is important to see how the computer sees this problem. A brute force solution or a naive solution demonstrates just that - trying to solve this problem without using human like intuition or heuristics.

                  A Flow Network

                  A classical Flow Network is a directed graph G with V, a set of vertices or couplings, and E is given set of edges between these vertices. A Flow Network has the following characteristics:

                  1. One source. In a flow network there should be only one source. If there are more than one sources they can be connected to one source using edges of infinite capacity and be reduced to a flow network.
                  2. One sink. In a flow network there should be only one source. If there are more than one sinks they can be connected to one source using edges of infinite capacity and be reduced to a flow network.
                  3. Edges are a subset of all possible connections between vertices, say u and v in V such that u != v. That is to say, it is not necessary for each vertex to be directly connected to every other vertex.
                  4. Each edge has a capacity, the maximum flow it can carry. We require this capacity to be a positive integer (int: ce > 0). We can flow any non-negative flow in any edge (0 <= fe <= ce).
                  5. There cannot be self-directed edges or reverse edges. i.e. no (u, u) edges in the network graph; and if (u, v) is in the graph, we cannot have (v, u). These are to avoid infinite loops in the search algorithm.

                  Example of a Flow Network (Cormen, et al.)

                  The following is a visual depiction of a network with six vertices and nine edges. Name of the vertex is inside the circle depicting it, and the capacity is marked on the edge. The direction of the edges are shown by an arrowhead.

                  1 is the source and 6 is the sink. Note that a source has no incoming edges, and a sink has no emerging edges, which is how these are defined.

                  Brute Force Search And Time Complexity

                  We can see that max flow out of 1 can be 29 (16 + 13) and max flow into 6 can be 24 (20 + 4). The upper limit to the max flow in this network is 24. But can we flow 24 across the network? In fact, the max flow through this network is 23. See if you can get to a max flow paths before reading on about the brute force search description below.

                  A brute force search for paths that start in 1 and end in 6 requires choosing among the possible ~29 permutations of the edges.

                  The possible paths between 1 and 6, and the max flow in these nine paths are shown in the table below:

                  Max FlowPath
                  1121, 2, 4, 6
                  2041, 2, 4, 3, 5, 6
                  3041, 2, 4, 3, 5, 4, 6
                  4041, 3, 5, 6
                  5041, 3, 2, 4, 6
                  6071, 3, 5, 4, 6
                  7041, 3, 2, 4, 3, 5, 6
                  8041, 3, 2, 4, 3, 5, 4, 6
                  9041, 3, 5, 4, 3, 2, 4, 6

                  As time consuming as it is, choosing only one of these paths is not the correct answer to the maximum flow problem. We need to search some more.

                  In the table you will notice that I have assigned a max flow to each path, while each edge in the path may have different caps. The flow in any path is limited to the capacity of the edge with lowest capacity on the path. These edges with the lowest capacity are called critical edges. Other than the critical edges, rest of the edges are not used to their full capacity.

                  Since the non-critical edges may be able to carry more flow, we may be able to find a parallel path going from source to sink through these non-critical edges. The brute force search for these parallel paths that allow for the highest flow-through requires sorting through the 29-1 possible combinations, to see which can co-exist and have the max flow. The time complexity of this search is O[2p].

                  The brute force search time to converge on the max possible flow in flow network is O[2m+2p]. Where, m is the no. of edges and p is the no. of unique paths from source to sink.

                  A much more elegant approach to the solution is the Ford-Fulkerson Method. It improves over the brute force method by using more efficient ways to search for the parallel paths, much closer to how a human may find these by visual inspection.

                  I started this discussion with an introduction to Flow Optimization. I discussed how this problem is an exponential time problem. Ford-Fulkerson method is a polynomial time solution to maximization of flow in a flow network. It comes close to how a human would visually look for paths in a flow network.

                  The complete code for Ford-Fulkerson using Breadth First Seach can be found here: Ford Fulkerson BFS Code (Python). This algorithm is a specific implementation of a general method described by Ford-Fulkerson and is called Edmond-Karp algorithm.

                  I will explain all parts of the code in the rest of the wiki, and I will recap the algorithm at the end.

                  Recall the flow network we are working with:

                  Python Code Description

                  Get The Flow Network

                  We can define the flow netwerk by defining the vertice and the connections between these vertices called edges. For convenience we define the vertices from 1 to n.

                  Each edges has a capacity, we store these in a one-to-one map called caps(E) in the code. I do the mapping using dictionaries in Python.

                  In the code I do this in the main function. def main(): n = 6 # no. of vertices m = 9 # no. of edges data = [(1, 2, 16), (1, 3, 13),\ (2, 4, 12),\ (3, 2, 4), (3, 5, 14),\ (4, 3, 9), (4, 6, 20),\ (5, 4, 7), (5, 6, 4)] edges = [] caps = {} for i in range(m): u, v, c = data[i] edges.append((u, v)) caps[u, v] = c

                  Initiate Search Variables

                  As we proceed in the algorithm, we will assign variable flows in edges. We store, track, and alter these these assignments in a map called flows(E'), where E' is a set of edges E along with reverse edges to allow flow cancellation.

                  This is one of the key innovation of Ford-Fulkerson. While network graph cannot have reverse edges, we allow flow in reverse edges as a mechanism to reduce the flow through an edge at a later stage of the algorithm, if needed for optimization of flow.

                  We define flows of edges and reverse edges in the very beginning of the function max_flow, where we do the BFS search. We will initiate the flows in all edges and reverse edges with 0.

                  def max_flow(edges, caps, n): source = 1 sink = n # initiate flows with 0 flows = {} for (u, v) in edges: flows[u, v] = 0 flows[v, u] = 0

                  As I mentioned in the introduction, while the critical edge on a path will be assigned its capacity of flow, the non-critical edges will have some left-over or residual capacity. We will keep track of the unused capacities after the flow is assigned in edges using a resid_caps map similar to caps.

                  resid_caps contains forward and reverse edges like flows. The capacity of the reverse edge is limited to the current flow in the forward edge.

                  We will make an adjacency graph based on resid_caps and call it resid_adj. If an edge reaches its capacity, has no residual capacity, it will not appear in the resid_adj. I also sort the neighbors in each vertex in reverse order of their residual capacity.

                  # initiate residual graph, by rules resid_adj = defaultdict(list) resid_caps = {} for (u, v) in edges: resid_caps[u, v] = caps[u, v] - flows[u, v] resid_caps[v, u] = flows[u, v] # cancellation if resid_caps[u, v] > 0: resid_adj[u].append(v) if resid_caps[v, u] > 0: resid_adj[v].append(u) for u in resid_adj.keys(): resid_adj[u].sort(key = lambda v: -1*resid_caps[u, v])

                  Search for Augmenting Paths in Residual Graph (Breadth First Search)

                  A path in resid_adj is called an augmenting path. Our goal will be to find in the resid_adj graph, the shortest path (BFS), with highest capacity (achieved by sorting neighbors by capacities).

                  The block from line 31 to 51 is the Breadth First Search. Note that in the search we will gather the path as edges rather than vertices to avoid adding any duplicates to the path. We will convert this edge to a list of vertices called final when we have found an augmenting path.

                  1. Initiate a queue as an empty deque object
                  2. The first edge we will visit will be the highest capacity edge out of source, (u, v).
                  3. Initiate the path as an empty list.
                  4. Add a tuple to the queue, which has the next edge and the path so far.
                  5. while there are elements in the queue, we do steps 6 - 12.
                  6. Pop (w, x) and path from the queue.
                  7. Append the edge (w, x) to thepath.
                  8. If the last vertex (x) does not have neighbors, we will pop the next item in the queue.
                  9. If x has neighbors, we call them y, we will explore these in a for loop (steps 9 - 13).
                  10. If y is sink, we will make final (list of vertices) out of path (list of edges).
                  11. We will terminate the for loop by calling break, and
                  12. we will set queue to an empty deque to break the while loop.
                  13. If y is not sink and (x, y) is not in path, we will add it, along with the present condition of the path to the queue. Note: I am making a copy of path as tmp, rather than adding the object path to the queue, since path is variable.

                  u = source v = resid_adj[source][0] queue = deque() queue.appendleft(((u, v), [])) # path = [] empty list while queue: (w, x), path = queue.pop() path.append((w, x)) if x not in resid_adj.keys(): continue for y in resid_adj[x]: if y == n: # reached sink path.append((x, y)) a = path[0][0] final = [a] for e in path: final.append(e[1]) queue = deque() break elif (x, y) not in path: tmp = [i for i in path] queue.appendleft(((x, y), tmp))

                  Once we find a path, we will allow the maximum possible flow in this path. This is limited by the edge on the path with the lowest capacity. As soon as we have assigned the flows in the edges on this path, the resid_caps and resid_adj will have to be modified. And we will try to find the next possible augmenting path in this residual graph. In every iteration some edges will get saturated, not allow any more flow, and soon we will not be able to get from source to sink. This is when our search will get terminated.

                  path = final.copy() final = [] if not path: print('All augmenting paths found. Terminating search') break else: # there exists a path between source and sink u = path[0] min_ = float('inf') for v in path[1:]: min_ = min(min_, resid_caps[u, v]) u = v f = min_ prev = max_flow max_flow += f u = path[0] for v in path[1:]: if (u, v) in edges: flows[u, v] += f resid_caps[u, v] -= f else: flows[v, u] -= f resid_caps[v, u] -= f u = v # update residual graph, tracker resid_adj = defaultdict(list) for u, v in resid_caps.keys(): if resid_caps[u, v] > 0: resid_adj[u].append(v) for u in resid_adj.keys(): resid_adj[u].sort(key = lambda v: -1*resid_caps[u, v])

                  Once the search terminates the code returns the max_flow and theflows assigned to edges to main. Main prints these out.

                  Recap And Final Comments

                  We can recap the search process as follows:

                  1. Initiate flows: assign a flow of 0 in all edges and reverse edges.
                  2. Initiate resid_caps map based on flows.
                  3. Initiate resid_adj map based on resid_caps.
                  4. Search for paths from source to sink using a queue.
                  5. While there is an augmenting path in resid_adj that leads from source to sink:
                    1. Find an augmenting path that leads from source to the sink such that we use any edge only once. Thus, we track the edges in resid_adj we have in the path.
                    2. Update flows for the edges in this path. The max flow possible in any path is limited by the smallest capacity edge. So, we will assign each edge a flow of min(resid_caps(ep)).
                    3. Update resid_caps based on flows.
                    4. Update resid_adj based on resid_caps.
                  6. The search terminates when there are no more feasible augmented paths between source and sink.
                  7. At this point the max flow can be evaluated as either the flow out of source or the flow in to sink.

                  Finding a path in a residual network can be done is a few ways:

                  • If we use Breadth First Search (BFS) we can find the shortest paths (path with least vertices) first, and have a good limit on the search time, but we will use a lot of memory.
                  • If we use Depth First Search (DFS) we will not use much memory at all, but the paths are not found in any order and the time could be worse than BFS.
                  • Asymptotically, BFS and DFS should take the same time. For a dense flow network, with lots of connections per vertex, DFS is a better choice. For a sparse flow network or where we are searching for shortest paths to reach a goal flow, and/or where memory is not the limiting concern, BFS will be a better option.

                  Let's consider the problem of finding all possible path from 1 to 6 in the flow network shown below.

                  One way to do this is a bread first search through 10_Adjacency Graphs. The code for this is provided here.

                  To begin we create an adjacency graph by going over each edge. adj = {} for (u, v) in edges: if u not in adj.keys(): adj[u] = [v] else: adj[u].append(v)

                  The adjacency graph of the network above is as follows:

                  { 1:[2, 3], 2:[4], 3:[2, 5], 4:[3, 6], 5:[4, 6] }

                  When we perform the search for all paths in this adjacency network. Initiate a list of paths as an empty list.

                  all_paths = []

                  Then we iterate over all the neighbors of the source (1) in a for loop.

                  A queue is a data structure where we add items to one side and extract them from the other side, so that we visit items in the order we arrived at them. In Python we accomplish this by using deque. It is imported from the module collections, where many important data structures are available. In this example we will keep track of the condition the next edge and the condition of the path using queue. In the first iteration the queue looks as follows:

                  next_edge, path

                  [1, 2), []

                  Inside the while loop we do the following steps.

                  • Pop the queue to receive w, x = (1, 2) and path = [].
                  • Add (1, 2) to the path. path = [(1, 2)].
                  • Then we search for an edge starting with 2, and add this vertex and the present version of the path to the queue. Queue becomes:

                  next_edge, path

                  (2, 4), [(1, 2)].

                  We also ensure that if we come across an edge already in the path, we ignore it

                  For example,

                  • The item we popped from the queue is ((5, 4), [(1, 2), (2, 4), (4, 3), (3, 5)]).
                  • We add (5, 4) to the path. path = [(1, 2), (2, 4), (4, 3), (3, 5), (5, 4)].
                  • Then we look among the neighbors of 4 for the next edge. We will ignore 3 since (4, 3) is already in the path. We will never add ((4, 3), [(1, 2), (2, 4), (4, 3), (3, 5), (5, 4)]) to the queue.
                  • We will go to the next neighbor of 4 and add ((4, 6), [(1, 2), (2, 4), (4, 3), (3, 5), (5, 4)]) to the queue instead.

                  If the edge ends in 6, the sink, we have found one path that goes from 1 to 6. We will not add this edge and path to the queue. Instead, we simplify the list of edges in the path to a list of vertices and add this to all_paths, the list of possible paths.

                  For example:

                  • if the path so far is [(1, 2), (2, 4), (4, 3), (3, 5), (5, 4)],
                  • we can add (4, 6) to the path. path = [(1, 2), (2, 4), (4, 3), (3, 5), (5, 4), (4, 6)].
                  • Then make a final sequence of vertices:
                  • a = path[0][0] a = 1.
                  • final = [a] final = [1].
                  • for e in path: final.append(e[1])
                  • final = [1, 2, 4, 3, 5, 4, 6].
                  • This is the version we append to all_paths.

                  We continue in this way until the queue becomes empty. When the queue becomes empty, we will have gathered all the paths starting at 1, 2 and ending in 6 in the adjacency graph.

                  Then we will do the next iteration of the for loop, which will start with 1, 3. Since 1 has only two neighbors in the example, this will end the search. In the code we return a list of all unique sequence of vertices that go from 1 to 6 in the network, which is printed from main().

                  [ [1, 2, 4, 6], [1, 2, 4, 3, 5, 6], [1, 2, 4, 3, 5, 4, 6], [1, 3, 5, 6], [1, 3, 2, 4, 6], [1, 3, 5, 4, 6], [1, 3, 2, 4, 3, 5, 6], [1, 3, 2, 4, 3, 5, 4, 6], [1, 3, 5, 4, 3, 2, 4, 6] ]
