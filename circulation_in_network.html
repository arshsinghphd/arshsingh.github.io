A graph G can be completely defined by vertices V; and the edges E, a subset of the possible ways of joining these vertices.

G[V,E]

An adjacency graph is a way of storing this information such that we can easily traverse across the graph. It is a map, a dictionary in Python, which stores a list of vertices that can be reached from any vertex, using the vertex as the key.

Breadth First Search

 28th June 2024 at 12:53pm
All

Let's consider the problem of finding a path from 1 to 6 in the flow network shown below.

One way to do this is a bread first search through Adjacency Graphs.

The adjacency graph of the network above is as follows:

{ 1:[2, 3], 2:[4], 3:[2, 5], 4:[3, 6], 5:[4, 6] }

When we do a breadth first search we start from 1. We also initiate a list of paths as an empty list.

Paths = []

Going forward if we arrive an any vertex we look for unvisited edges and add them to a data structure called queue, where we add items to one side and extract them from the other side, so that we visit items in the order we arrived at them.

In this example we will keep track of the condition of the path and the next vertex. In the first iteration the queue looks as follows:

[(3, [1]), (2, [1])].

Then we extract 2 from the queue and add its neighbors to the queue. Queue becomes

[(4, [1, 2]), (3, [1])].

We also ensure that if we come across an item which will repeat an edge in the path, we ignore it and go to the next item in the queue. For example, if the item we popped from the queue is

(3, [1, 2, 4, 3, 5, 4]),

we will ignore it since (4, 3) is already in the path.

We continue like this until we arrive at an item which ends in 6. We will add this path to Paths, the list of possible paths, and continue until the queue becomes empty.

When the queue becomes empty, we will have gathered all the paths starting at 1 and ending in 6 in the adjacency graph.

Ford-Fulkerson For Circulation

 10th July 2024 at 1:58pm
Index

I started with general information about flow optimization in Introduction to Flow Optimization, followed by discussion about the Ford-Fulkerson Method.

I discussed the problem statement of circulation and the key ideas hidden in the problem that will drive this algorithm in Max Flow Vs. Circulation.

In this wiki, I will modify the Ford-Fulkerson method for the circulation problem. Before I go on, let's discuss how we can tackle the issue of parallel and/or reverse edges in a network.

Parallel Edges, Reverse Edges

Like in the flow network for flow maximization, copies or reverse edges are not allowed. If a copy, or a reverse, of an edge already in the list is encountered, these can be replace these with an equivalent: two connected edges of exactly the same le and me, passing through a new vertex. That is, (u, v) is replaced with (u, v'), (v', v), where v' is a new vertex.

To ensure that the order of the edges is unchanged, to print the assigned flow out in the same order, the new edges are added at the end of the list of edges.

Modified Ford-Fulkerson

From the given data, two lists are created: Edges, a list of edges, and an associated list, Caps, of capacities: lower le and upper me. These lists are matched by their position in the lists. These are modified if parallel or reverse edges are encountered as discussed earlier.

A map flows, is created keyed with edge e in Edges, to fe, the flow in this edge. This map is initiated with 0. Note that unlike the max flow, in the circulation problem reverse edges are not needed.

BFS For Critical Egdes

As discussed in Max Flow Vs. Circulation, the key point in this algorithm is that any edges that needs a minimum flow le > 0, must lie on a cycle. These are called critical edges in this wiki. It follows logically from the fact that there are no source of flow or sink to absorb flow, and flow across vertices is conserved. Program runs faster if critical edges are sorted in reverse order of their le.

Based on flows a map, resid_caps, of edges to their residual capacity, is created. Based on residual_cap, an adjacency graph is created, resid_adj, mapping the vertices of edges with residual capacity if > 0.

Augmented paths in the flow maximization problem means any viable path in the residual graph from source to sink. In circulation, these are replaced by augmented cycles in residual graph that goes from one of the critical edges, back to it.

The main algorithm consists of two nested while loops:

  • Outer while loop makes sure all the critical edges lie on a cycle.
  • Inner while loop is a Breadth First Search, BFS, for a cycle starting from a yet not discovered critical edges, say (u, v).

Notes:

  1. When implemented with BFS, Ford-Fulkerson, it is also knows as Edmonds-Karp algorithm. The runtime of this algorithm is O(VE2).
  2. Sinks may be encountered in BFS, vertices that do not have edges going out, and thus no neighbors.
  3. Two queues are maintained in the inner loop, one that houses the vertices to explore and another that houses the path till the vertex in the queue. While the two can be combined into one, such an object takes up too much memory in practice.

DFS With Backtracking

If memory is a bigger concern than time, DFS with backtracking can be used. DFS does not maintain any queues. When implemented smartly, it can work with two lists: path and edges tracker; and can be very memory efficient. The runtime of this algorithm is still O(VE2).

While asymptotically both BFS and DFS with backtracking work in same time complexity, DFS works slower in practice.

Termination

The inner loop may terminate in two ways: when the queue is empty or when an edge that ends with u, (t, u), is discovered, making this a cycle.

  • If the queue is empty, and the discovered path is not a cycle, the critical edge does not lie on any cycle and there cannot be circulation in this graph. The search can be terminated at this point.
  • If a cycle is found, the following steps are performed in order:
    1. All the edges on the cycle are traversed, marking all the critical edges discovered on it, and noting le and resid_caps[e] for each of them;
    2. Flows for each edge in the cycle is updated with minimum permissible flow in the cycle: max(le) or min(resid_caps[e]), over all e in Edges, whichever is lesser;
    3. resid_caps and resid_adj for these are updated with fe; and finally,
    4. a yet undiscovered critical edge is found, and becomes the starting point for the next iteration of the inner loop.

The outer loop is terminated when all critical edges are discovered.

There is only one way to get to NO, if it is discovered that any of the critical edges does not lie on a cycle; and only one way we can get to a YES: if all critical edges are discovered to lie on some augmented cycle.

Doesn't Pass Tests For Capstone

The Edmonds-Karp algorithm with BFS exceeds both the time- and memory- limits. The DFS with backtracking exceeds time-limits.

To reduce the time I will dabble with Push and Relabel Method.

Ford-Fulkerson Method

 10th July 2024 at 2:00pm
Index

Ford-Fulkerth method for maximization of flow is a polynomial-time solution to a pretty complex problem.

It involves creating a new graph and map based on the flow network.

Let's call the flow network G(V, E) with an associated map Caps(E).

As we proceed in the algorithm, we will assign variable flows in edges. We store, track, and alter these these assignments in a map called Flows(V,E'), where E' is a set of possible edges u*v for every u, v in V such that u != v.

This is the key innovation of Ford-Fulkerth. While network graph cannot have reverse edges, flow keeps track of flow in reverse edges, since it may be required to reduce the flow in an edges for optimization at a later stage of the algorithm.

We will also keep track of the unused capacities after the flow is assigned in edges using a Residual Caps map. Like flows, this graph also contains reverse edges. The capacity of the reverse edge is limited to the current flow in the forward edge, as a mechanism of cancellation of flow in the network edges down to 0 if needed at a later stage, for optimization.

Let's call the residual capacity map R defined over (V,E'), where E' is as defined before, with the limitation that we remove any key:value pair if the residual capacity is 0, i.e there cannot be any more flow in that edge. Note that the reverse edge will surely be in the graph if the cap. of the edges > 0.

R(V, E': if flow[e] > 0 for each e in E').

For convenience we will make an adjacency graph based on Residual Caps and call it Residual Graph.

Since an edge can be used in a path and still have some capacity left to be used in another path later on, we differentiate the path in Residual Graph from other graphs (where we can use an edge only once) by calling them augmenting path.

The method proceeds as follows:

  1. Initiate Flows: assign a flow of 0 in all edges and reverse edges.
  2. Initiate Residual Caps map based on Flows.
  3. Initiate Residual Graph map based on Residual Caps.
  4. Initiate a variable max flow with 0. Now, we search for paths from source to sink.
  5. While there is an augmenting path in Residual Graph that leads from source to sink:
    1. Find an augmenting path that leads from source to the sink such that we use any edge only once. Thus, we track the edges in Residual Graph we have in the path.
      • Finding a path can be done is a few ways.
      • If we use Breadth First Search we can find the shortest paths (path with least vertices) first, and have a good limit on the search time.
    2. Update Flows for the edges in this path. The max flow possible in any path is limited by the smallest capacity edge. So, we will assign each edge a flow of min(residual_capacity[e'], e' in path).
    3. Update Residual Caps based on Flows.
    4. Update Residual Graphs based on Residual Caps.
  6. The search terminates when there are no more feasible augmented paths between source and sink. At this point the max flow can be evaluated as the either the flow out of source or the flow in to sink. Asserting that these are the same can be a good test for the correctness of the algorithm as well.

Introduction to Flow Optimization

 14th July 2024 at 11:02am
Index

The classical problem of Maximum Flow is an optimization problem of optimizing the flow through a Flow Network from a source to a sink.

Visualize of a network of pipes connected via multi-sized couplings with one tap and one outlet. While we know the capacity of each pipe (based on its diameter, thickness of the walls, material of construction etc.), if we try to flow more than max. possible we will disrupt the system, causing leakages at joints, burst pipes etc. It seems like a simple problem but is not.

Playing around with a toy example can show how if we use brute force, this seemingly simple problem has exponential time-complexity of O[2m], where m = len(E), where E is the set of edges in the network. And even then, we may not converge to the correct solution.

This problem has many variations used for solving many seemingly unrelated real world problems. One of these is circulation in a network which is used for deciding the multiplicity of k-mers in genome assemblers. Other examples include planning for road traffic signals, and air-traffic-control.

Definition of a Flow Network

A classical Flow Network is a directed graph G(V, E), where V is a set of vertices and E is given set of edges between v of V. E is a subset of P(V, 2), all possible permutations of size 2 in V. A Flow Network has the following characteristics:

  1. One source. if there are more than one sources they can be connected to one source using edges of infinite capacity and be reduced to a flow network.
  2. One sink. if there are more than one sinks they can be connected to one source using edges of infinite capacity and be reduced to a flow network.
  3. Each edge has a capacity, the maximum flow it can carry. We require this capacity to be a positive integer.
  4. There cannot be self-directed edges or reverse edges. i.e. no (u, u) edges in the network graph; and if (u, v) is in the graph, we cannot have (v, u). These are to avoid infinite loops in the search algorithm.

One classical problem is maximization of flow in flow networks, and a classical solution is using Ford-Fulkerson Method.

Example (Cormen, et al.)

The following is a visual depiction of a network with six vertices, 1 is the source and 6 is the sink, and nine edges. Name of the vertex is inside the circle depicting it, and the capacity of the edge joining two vertices is marked close to the line joining them.

We can see that max flow out of 1 can be 29 (16 + 13) and max flow into 6 can be 24 (20 + 4). The upper limit to the max flow in this network is 24. But can we flow 24 across the network? In fact, the max flow through this network is 23. See if you can get to it using brute force search described below.

A path is defined as an order of edges that do not traverse any edges only once. A brute force search for paths that start in 1 and end in 6 requires choosing among the possible permutations of the edges (factorial[(29-1)-1]). This can be achieved much more easily, in polynomial time, using Adjacency Graphs.

The possible paths between 1 and 6, and the max flow in these path are shown in the table below:

As time consuming as it is, choosing only one of these paths is not the correct answer. There is more searching to do to get to maximum possible flow.

The flow in any path is limited to the minimum flow in any of the edges on the path. These edges with the lowest capacity are called critical edges. Other than the critical edges, most of the edges are not used to their full capacity. The non-critical edges can be re-used in other paths and thus some of these paths can exist parallel to each other. The brute force search for these parallel paths that allow for the highest flow-through requires choosing from the 28-1 possible combinations among these paths.

The brute force search time to converge on the max possible flow in flow network is O[2m+2p]. Where, m is the no. of edges and p is the no. of unique paths from 1 to 6.

A much more elegant approach to the solution is the Ford-Fulkerson Method.

Max Flow Vs. Circulation

 12th July 2024 at 7:09pm
Index

Problem

The circulation problem is a special case of the Flow Problem discussed in Introduction to Flow Optimization.

In circulation problem there are no sources or sinks. And other than the capacity, me, the edges also have a (required) minimum flow le (>= 0). The task is to check if we can assign flows in the edges, fe, such that:

le <= fe <= me;

while maintaining conservation of flow across v in V, such that:

SUM( f(u, v) : u in V) = SUM( f(v, w) : w in V).

The problem prompt hints that we can modify the Ford-Fulkerson Method (F-F) to be able to do this task, but the modified F-F times out for case # 20. I will also show how to modify Push and Relabel Method for circulation in Push and Relabel For Circulation. Asymptotically P-R-F runs faster than F-F.

Other Parameters

The edges provided are not self-loops, but can be parallel to other edges. i.e. there are be two edges (u, v) with same or different le and me.

We are also given the following parameters:

  • Number of vertices, n: 2 <= n <= 40.
  • Number of edges, m: 1 <= m <= 1600.
  • 0 <= le<= me <= 50.

How it is Used in Genome Assembler

Let's say we got many reads from a genome. For now we can also assume these reads are free of error. But the reads (all of length n) are not a complete n-mer cover of the genome. To ensure a complete cover, we can break each read into many shorter parts each of length k. This theoretically can provide us a complete kmer cover, but now we may have many copies of some of the kmers. How many copies of kmers are genuine part of the genome? We can answer this question by answering an analogous problem the circulation problem. In the circulation problem we are trying to find out the flow in each path. In the kmers problem we can think that each kmer can be converted into an edge of the deBruin graph and has an le of 1. The final flow is analogous to the number of times we need to use a kmer to be able to put all the contigs together into a circular genome.

Key Ideas Hidden in the Problem

  1. If paths are not cycles, their first and last vertices become a source and a sink respectively. Conservation of flow requires that each source and sink must have le of 0. If this condition is not met, there cannot be circulation in this network.
  2. Corollary to the point above, any edge with le > 0, let's call these critical edges, must lie on a cycle. In other words, we should be able to find a path that starts and ends at each critical edge. If no such cycle exists, the network cannot have a circulation.

The algorithm to solve the ciculation problem is described in Ford-Fulkerson For Circulation.

Push and Relabel Method

 13th July 2024 at 6:18pm
Index

The Push and Relabel to Front (P-R-F) with method is an improvement over the Edmonds-Karp (E-K) algorithm, the BFS version of Ford-Fulkerson method. The asymptotic run time of E-K is O(VE2), while the P-R-F runs in O(V3), always better than E-K.

Intuition

The intuition of Push and Relabel is different in the way vertices are imagine in E-K.

The intuition of E-K is parallel streams of flow in a network of edges connected at vertices. A vertex in P-R-F is not only a junction but is also connected to a reservoir that can store potentially infinite amounts. These vertices and reservoirs are sitting on a variable height pole. Flow can only occur from a higher to lower vertex, and we can adjust the height of a vertex if we need to make flow occur.

As the algorithm progresses we will push as much flow as we can from the source towards the sink. When we are done with this forward pushing, sink will receive as much flow as it can, which is the maximum flow in the network. Some of the vertices will have excess flow, which we will push backwards. When no vertices have any excess flow we will have found the flows in edges that lead to the maximum flow in the network. It is actually that simple. The implementation needs a few nifty tricks but is not overly complicated. If not clear now, the will become clear in the following paragraphs.

Initialization:

Let's recall the flow network:

We are given a list of edges with their capacities. It is useful to store the capacities as a map linked to the edges, for faster recall.

The source is 1 and the sink is 6.

We can initialize by assigning height[source] = len(edges) - 1 and 0 for all the other vertices.

We start off flows by assigning the edges connected source to their neighbors to the capacities of these edges. Flow of all other edges are assigned to 0. This is possible since the height of source > 0 (height of the neighbors initially).

Relabel and Push

After the initialization, we have excess flow at vertices 2 and 3.

Consider 2. We will push the excess flow from 2 towards its neighbor 4.

To accomplish this, we will have to ensure that the height of 2 is above 4. This is relabeling. Then we will push the excess from 2 to 4: flow as much flow from 2 to 4 as the edge (2, 4) will allow, and adjusting all the variables this impacts. The heart of P-R-F are these two operations: relabel and push.

Relabel

If there is excess flow at u, residual capacity in the edge (u, v) and we want a flow from u to v, the height of u has to be at least 1 above the height of v. If it is not the case, we can relabel u's height:

height[u] = 1 + height[v].

Forward Pushing

For P-R-F we will have to maintain the following:

For vertices:

  1. heights: a mutable list of heights of vertices, indexed by vertices.
  2. excess: a list of excess at vertices, indexed by vertices.
  3. resid_adj: a map of vertices to a list of its neighbors in the network which have residual capacities. This is based on resid_caps of edges. Sort the list by reverse of resid_caps for efficiency and correctness.
  4. resid_adj_reverse: a map of vertices to a list of its neighbors in the reverse network, which have residual capacities. This is based on resid_caps of reverse edges. Sort the list by reverse of resid_caps for efficiency and correctness.

For edges: we are given a list of capacities.

  1. flows: a map of flow in edge to the current flow in the edge,we allow flow in reverse edges, before returning the flows, we can carefully subtract the flow in reverse edges from the flow in edges and remove the reverse edges from the map. Or one cannot have reverse edges in this map, maintaining the flow of reverse edges by subtracting from the flow of edges.
  2. resid_caps: a map of edges, and reverse edges, to their (mutable) residual capacities. Like in E-K, we will allow a reverse edge to have the residual capacity to of the flow in the (forward) edge.

Algorithm

Now that I have explained what the algorithm does intuitively, I will write a code in python for the relabel and push method. We ensure that the vertices are visited in topological order by using a queue and we maintain the adjacency graph sorted by reverse of residual capacities that ensures efficiency and correctness.
